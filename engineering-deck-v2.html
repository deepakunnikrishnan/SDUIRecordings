Problem

UI is compiled into binaries. That's the bug.

Every layout decision, conditional, and feature flag is baked at build time. Changing anything means a full release pipeline. We move those decisions server-side — one typed JSON manifest per bank, resolved at request time.

Three Integration Modes

One SDK. Three call sites. Zero business logic in the client.

// Mode 1 — full-screen Activity (bank picker tab)
startActivity(CardManagementActivity.createIntent(ctx, bankId, colors))

// Mode 2 — screen embedded in host nav (Cards tab)
CardManagementScreen(
    overrideBffClient = BffClient(url, mapOf("X-Bank-Id" to bankId)),
    overrideColors    = bank.colors,
    viewModelKey      = bankId,
)

// Mode 3 — single slot widget (Home tab)
key(bankId) {
    CardManagementWidget(
        slotId             = bank.widgetSlotId,   // "hero" | "accountBanner" | "portfolioHeader"
        overrideBffClient  = BffClient(url, mapOf("X-Bank-Id" to bankId)),
        overrideColors     = bank.colors,
        widgetViewModelKey = bankId,
    )
}
Architecture

Five layers. Clients know nothing.

Host App (Compose) ─── 3-tab demo: Launch / Cards / Home
  └── Card Management SDK
        ├── Mode 1: CardManagementActivity  (Intent extras: bankId + 8 color Ints)
        ├── Mode 2: CardManagementScreen    (override params: bffClient, colors, vmKey)
        └── Mode 3: CardManagementWidget   (override params: bffClient, colors, vmKey)
              └── android-sdui engine
                    └── BffClient(url, headers)  ──► BFF  X-Bank-Id: meridianbank
                                                        └── Compositor
                                                              ├── manifest-resolver
                                                              │     └── data/meridianbank/card-dashboard-manifest.json
                                                              ├── data-coordinator(bankId)
                                                              │     └── card-service, txn-service (bank-keyed data)
                                                              └── slot-assembler (transforms, paramsHash)
Bank Routing

X-Bank-Id routes the full stack. One header, three different screens.

  • BFF reads X-Bank-Id header → threads bankId? into assemble(screenId, slotId, bankId)
  • resolveManifest tries data/{bankId}/card-dashboard-manifest.json first, silently falls back to default
  • Mock services receive bankId? → bank-keyed card data (card_123 / card_456 / card_789) and transactions
  • NovaPay: no header sent → default manifest (hero + controls + list)
  • Apex: X-Bank-Id: apexbank → account-banner + activity-feed, no card controls slot
  • Meridian: X-Bank-Id: meridianbank → portfolio-header + card-identity + controls + activity-feed
ViewModel Isolation

key(bankId) alone is not enough. Here's why.

key(bankId) changes Compose composition identity — it tears down and rebuilds the subtree. But the ViewModel is stored in the Activity's ViewModelStore under a class-name key, outside the composition tree. On the next bank switch the old VM (with the stale BffClient) is returned from cache.

// ❌ broken — VM reused across banks
key(bankId) {
    SduiWidget(
        bffClient = bankClient
        // no viewModelKey
    )
}
// ✅ correct — VM keyed per bank
key(bankId) {
    SduiWidget(
        bffClient    = bankClient,
        viewModelKey = bankId,   // ← new param
    )
}
Content Management Loop

Edit JSON → pull to refresh → layout updates live. No app rebuild.

  • Server manifest is a JSON file — add a slot, rename a label, reorder sections
  • No recompile, no deploy pipeline, no store review needed
  • Pull-to-refresh in Activity, Screen, and Widget mode all hit the same BFF endpoint
  • Each mode gets independent pull-to-refresh; Widget refresh is slot-scoped (not the whole screen)
  • Phase 2: swap file with Postgres-backed Manifest Service — same loop, persistent and versioned
Phase 2 Roadmap

The prototype proves the loop. Phase 2 makes it production-ready.

  • Manifest Service: Postgres JSONB, admin API, optimistic concurrency, versioned rollback
  • Compositor as a service: Redis pub-sub cache invalidation, horizontal scale
  • App entity: one config root per bank — routes, auth contract, theme, embed policy, action scope
  • Universal action schema: idempotency keys, all action types (nav, alert, state, deeplink, sequence)
  • iOS renderer: SwiftUI, same manifest format — CompositeDefinition is platform-agnostic
  • Portal: drag-and-drop manifest editor — PM edits layout without touching JSON
